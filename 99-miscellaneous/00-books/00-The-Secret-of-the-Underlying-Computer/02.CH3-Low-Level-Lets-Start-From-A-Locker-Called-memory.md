# CH3: The Low-Level Layer: Let’s Start with Memory as a Set of Lockers

## 3.1 The Nature of Memory, Pointers, and References

### 3.1.1 What Is the Nature of Memory? Lockers, Bits, Bytes, and Objects
- Memory consists of memory cells that store bits composed of 0s and 1s
- Eight bits are grouped together to form one byte, which is used as a unit to represent more information
- Every byte has its own address in memory (memory address)
- A single memory address can be used to locate multiple memory cells (addressing)
- With 8 bits (1 byte), there are 2^8 = 256 possible combinations, allowing up to 255 unsigned integers, which is still quite limited
- Therefore, it is common to use 4 bytes together to represent one integer (2^32 ≈ 4 billion)

### 3.1.2 From Memory to Variables: The Meaning of Variables
- Values are stored in memory and read back, while computations are performed in registers
```nasm
store $1 6 -> store 1 in memory 6
load r1 6 -> read value in memory 6 to registry 1
```
- It is also possible to copy a value stored in one memory location and move it to another memory location. However, what if the data is large, such as a structure?

### 3.1.3 From Variables to Pointers: Understanding Pointers
- Suppose a structure is stored in a, and the memory address of a is "3". Instead of copying the entire structure into another variable b, it is more memory-efficient to store "3" and access the value through it
- Pointer = a variable suitable for storing a memory address (an abstraction of memory addresses at a higher level)
```nasm
load r1 1 -> read value in memory 1 (which is memory addr of b, so the value is "3") to registry 1. This is incorrect syntax
load r1 @1 -> read memory address in memory 1 to registry 1. This is correct syntax
```
- Address 1 → Address 3 → Data: this process is called indirect addressing
- Assembly language does not have the concept of variables, so understanding this layer is important
- In high-level languages, unlike assembly, there is no need to know the address of variable b
- b → data: this is why pointers are considered a higher-level abstraction

### 3.1.4 The Power and Destructiveness of Pointers: Ability and Responsibility
- With pointers, programmers can directly manipulate hardware such as memory because they can directly access memory addresses
- This is impossible in languages that do not support pointers

### 3.1.5 From Pointers to References: Hiding Memory Addresses
- Languages that provide references instead of pointers do not allow access to memory addresses and cannot perform pointer-like operations, but they avoid copying data and achieve the same effect as using pointers
- In other words, a reference is a further abstraction on top of a pointer

## 3.2 What Does a Process Look Like in Memory?

```nasm
---------
Kernel
--------- 2^48 - 1
Stack Area
--------- top of stack
v
v
v
^
^
^
---------
Heap Area
---------
Data Area
---------
Code Area
--------- 0x400000

--------- 0
```

- Code and data areas → created when the executable file is initialized
- Heap area → used for dynamic memory allocation
- Stack area → used for function calls

### 3.2.1 Virtual Memory: What You See Is Not Always the Reality
- If memory really looked like the structure above, wouldn’t every process’s code area start at the same address and `malloc` return the same starting address?
- No, this is not a problem. These addresses are virtual addresses. Before real memory operations occur, they are translated into physical addresses
- Therefore the structure above is only a virtual view; real physical memory rarely looks like this
- Real physical memory looks more like this:

```nasm
---------
---------
Data Area 1
---------
Code Area 1
---------
---------
Heap Area 2
---------
---------
Stack Area 1
---------
---------
---------
Heap Area 3
---------
---------
---------
Heap Area 1
---------
Code Area 2
---------
---------
```

- A process is divided into equal-sized pieces and stored in memory (for example Heap 1, 2, and 3 all have the same size)
- These pieces are scattered randomly across physical memory
- Virtual memory makes this scattered layout appear as a continuous address space → this is possible because of the mapping relationship between virtual memory and physical memory
- This is why the page table is important

### 3.2.2 Pages and Page Tables: From Virtual to Physical
- The structure that maintains this mapping relationship is called the page table
- Each process has its own page table
- The pieces of a process are called pages

### 3.3.3 Stack Frames and the Stack Area: A Macro Perspective
- Every function creates its own stack frame (or call stack) when it executes
- In a process stack area, higher addresses are at the top and the stack grows toward lower addresses

### 3.3.4 How Are Function Jumps and Returns Implemented?
- When function A calls function B, control transfers from A to B
- Control means which function’s machine instructions the CPU is currently executing
- Two pieces of information are needed when control transfers
    - Return → where execution came from (for example where A stopped)
    - Jump → where execution goes (the address of B’s first instruction)

```nasm
Machine instructions of function B
0x400540
0x400541
0x400542 ret

Machine instructions of function A
0x400563
0x400564
0x400565 call 0x400540
0x400566
```

- When the CPU executes `0x400565`, it performs the call instruction
- Before jumping to `0x400540`, the CPU pushes the next instruction address `0x400566` onto the stack
- This saves the location to return to after B finishes executing

```nasm
-----------

// Stack frame of function A

-----------
0x400566
-----------

// Stack frame of function B

-----------
```

### 3.3.5 How Are Parameters Passed and Return Values Retrieved?
- In x86-64 most parameters and return values are passed through registers
- When A calls B, A places parameters in registers
- B reads the parameter values from those registers
- If registers are insufficient, extra parameters are stored in the stack frame

```nasm
------------

// Stack frame of function A

-----------
Parameter 1
-----------
... Parameter N
-----------
0x400566
-----------

// Stack frame of function B

-----------
```

### 3.3.6 Where Are Local Variables?
- Local variables are variables defined inside a function
- They can be stored in registers, but if there are more variables than registers they are stored in the stack frame

```nasm
------------

// Stack frame of function A

-----------
Local variables
-----------
Parameter 1
-----------
... Parameter N
-----------
0x400566
-----------

// Stack frame of function B

-----------
```

- Registers are CPU internal resources. If functions A and B use the same registers, values could be overwritten

### 3.3.7 Saving and Restoring Registers
- Before using registers for local variables, the original values must be saved
- After the function finishes, those values must be restored

```nasm
------------

// Stack frame of function A

-----------
Saved register values
-----------
Local variables
-----------
Parameter 1
-----------
... Parameter N
-----------
0x400566
-----------  --> end of A stack frame

// Stack frame of function B

-----------
```

### 3.3.8 Seeing the Big Picture
- The stack area has a limited size. If this limit is exceeded, a stack overflow occurs

## 3.4 힙 영역: 메모리의 동적 할당은 어떻게 구현될까? 

### 3.4.1 힙 영역이 필요한 이유 
- 전역 변수가 아니지만 여러 함수에 걸쳐 사용할 수 있는 데이터가 필요할 때 힙 영역 활용
- 프로그래머가 직접 관리하는 메모리 영역으로, 데이터 저장에 필요한 메모리 양과 사용 완료 후 무효화 작업까지 담당함
- C/C++ 같은 언어에서는 malloc 또는 new 예약어 사용하여 할당, free 나 delete 예약어를 사용하여 반환

### 3.4.2 malloc 메모리 할당자 직접 구현하기
- C/C++ 에서 메모리의 동적 할당과 해제 요청은 모두 전문적으로 힙 영역에 메모리를 할당하고 해제하는 작은 프로그램에 전달되어 처리 -> malloc 메모리 할당자
- 따라서 프로그래머는 직접 메모리 할당 구현 필요 없이 malloc 활용 가능
- malloc을 직접 구현해 본다면?
    - 메모리 할당자는 적절한 크기의메모리 영역을 제공하기만 하면 됨 
    - 어떤 내용을 저장할지는 무관 
- 힙 영역은 거대한 배열과 같음 

### 3.4.3 주차장에서 메모리 관리까지
- 주차장의 주차 공간이 비어 있거나 주차되어 있는 것처럼 메모리의 공간도 비어 있거나 이미 할당되어 있음

    1. 어떤 공간이 비어 있나?
